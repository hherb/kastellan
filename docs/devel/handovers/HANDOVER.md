**✅ MERGED (2026-07-20): [#471](https://github.com/hherb/kastellan/issues/471) — the guest kernel was fetched unchecked, and a bad copy was then reused forever. PR [#478](https://github.com/hherb/kastellan/pull/478), merged to `main` as `dd10bd68`.** Every `build-*-rootfs.sh` did `[ -f "$OUT_DIR/vmlinux" ] || curl … "$KERNEL_URL"`: version-pinned but never integrity-checked, so a compromised mirror/MITM/wire corruption hands us an arbitrary kernel that boots **inside** the containment boundary (the micro-VM *is* the boundary; the kernel enforces it). **The `[ -f … ] ||` guard is the part that actually mattered** — whatever landed there first was reused unchecked by every later build, making a one-time compromise permanent. New `scripts/workers/microvm/lib/guest-kernel.sh` is the single place the URL, arch table and sums are written; all 8 scripts (7 in `microvm/`, kv-demo one dir across) now call `fetch_guest_kernel "$OUT_DIR"`. One shared file rather than 8 parallel edits **on purpose** — 8 copies is 8 chances to drift, the [[#475]] shape, except here the drift would be a *silently unverified download*. **Three properties:** an already-present kernel is verified not trusted; a rejected kernel is moved to `vmlinux.rejected.<sha256>` rather than deleted (the build stops and the next run starts clean, but "what did we almost boot?" stays answerable); the download lands on a per-process `vmlinux.partial.<pid>` and is renamed only after it verifies, so an interrupted run can't leave a trusted-looking file. **On the sums — honest limitation:** recording them from a download is trust-on-first-use; upstream publishes no signature for this CI bucket, so TOFU is the ceiling. aarch64 has a **second witness with three weeks of temporal separation** (it matches the DGX's working `vmlinux`, independently fetched 2026-06-27 and used for every VM run since — a substitution would have had to be in place the whole time); **x86_64 was re-fetched 2026-07-20 from two hosts on separate network paths** (DGX + dev Mac, both matching), which kills the local-MITM and mis-recorded-sum cases but leaves it origin-level TOFU with no temporal separation. Documented in the snippet, not left implicit. **Verified — DGX:** `cargo test --workspace` **2607 / 0 / 50** (+10 = exactly the new tests), clippy clean, 0 `[SKIP]`; the DGX's **live** kernel verifies against the pin, so no deployment breakage. macOS: all 17 `microvm::tests` pass — the new tests are deliberately **not** `cfg(linux)`, since a fail-closed check exercised on one host is half-verified. **Both negative cases checked:** corrupting the recorded sum makes the *genuine* kernel be rejected (the comparison is load-bearing, not a rubber stamp), and reintroducing a stray `KERNEL_URL` fails the drift pin by name. **Post-review pass (3 findings, all fixed + pinned by a test each):** (1) the quarantine `mv`'s exit status was unchecked, so on a read-only `OUT_DIR` the build printed `quarantined: …` naming a file that was never written — it still failed closed, but *reported an evidence trail that did not exist*, which is the same "fails/reports for the wrong reason" class as the zsh `path` bug below; now the failure is reported and the false claim suppressed. (2) Evidence was written to a fixed `vmlinux.rejected`, so a second bad kernel **destroyed the first** — the whole point of keeping the bytes is answering "what did we almost boot?", and that is worth little if only the newest attempt survives; now content-addressed `vmlinux.rejected.<sha256>`, which also makes re-rejecting identical bytes idempotent instead of piling up corpses. (3) `vmlinux.partial` was a fixed name, so two build scripts running at once scribbled over each other's in-flight download — harmless (the sum caught it) but it produced a *baffling spurious rejection*; now per-process `.partial.<pid>`, at the honest cost that a hard-killed run leaves its temp behind (deliberately not swept: this function cannot tell a corpse from another live build's download). **Also from the review: these unit tests now run in CI.** `linux-check.yml` gained a `cargo test -p kastellan-tests-common --locked` step — the crate's tests are hermetic (no PG, no bwrap, no KVM, no network, ~0.1 s), and `kernel_pin_is_the_only_place_the_kernel_url_appears` only pays off if it fires on a **PR**; leaving a drift guard to an operator's occasional DGX run means the drift it guards against is exactly the case least likely to be caught. `cargo test` needs real codegen where `check`/`clippy` do not, so the step costs more than the two above — accepted deliberately for a security guard on every PR. **New dual-platform trap, caught in self-review before it shipped:** the three new tests counted evidence files with `ls … | wc -l` and asserted `contains("kept=1")`. GNU `wc` prints `1`; **BSD/macOS `wc` pads to `       1`**, so the assertion matched on the DGX and would have failed on the Mac — a green Linux run hiding a Mac break, the *mirror* of [[cfg-linux-e2e-deadcode-dgx-clippy]] and the same shape as #475's post-review addendum. Fixed with `| tr -d ' '`. Generalised lesson: **shelling out from a test makes coreutils output-format differences a portability surface**, not just `cfg` gating — if a test greps or compares text produced by `wc`/`stat`/`date`/`sed`, it needs both hosts before it is believed. **Gotcha worth keeping:** `verify_sha256`'s local was named `path`, which **in zsh is the array tied to `$PATH`** — sourcing from an interactive zsh destroyed command lookup, so it reported "no hasher" instead of the mismatch it was asked about. Fail-closed, but for the wrong reason; renamed to `file`. Production is bash and was unaffected. **Out of scope:** `install-firecracker.sh` still fetches the Firecracker binary unchecked — that is [#386](https://github.com/hherb/kastellan/issues/386), which can now reuse `verify_sha256`. **New follow-up lodged from the review: [#479](https://github.com/hherb/kastellan/issues/479) — verification is build-time only.** `sandbox/src/linux_firecracker.rs:64` boots `dir.join("vmlinux")` with nothing re-checking it, and `/var/lib/kastellan/microvm/` is owned + writable by the agent's own OS user — *precisely* what `docs/threat-model.md` assumes a worst-case compromise reaches. So an attacker at that level can replace the kernel and every later VM boots it; no build need ever run again for this to hold. Two candidate fixes in the issue (hash at boot; or make the dir root-owned so the write primitive is gone) — the second is the stronger one. This is a bigger surface than #471 closed, and was kept out of this PR so the two review independently.

## Key design decisions locked in

- **Vendor-neutral, AGPL-compatible deps only.** AGPL project; all third-party deps must be AGPL-compatible (Apache-2.0, MIT, BSD, MPL, LGPL, (A)GPL all fine).
- **Cross-platform first-class.** Linux (DGX Spark primary) + macOS (Apple Silicon and Intel). No Linux-only code without a macOS counterpart of equivalent guarantee.
- **Rust core, Python workers.** Rust for core (no eval/dynamic surface); Python only inside sandboxed tool workers. shell-exec is Rust because it's a thin execve wrapper — Python's first appearance will be `python-exec` in Phase 4 (or possibly `web-fetch` earlier).
- **Hybrid LLM with policy routing.** Local-first via OpenAI-compatible HTTP (vLLM/SGLang on Linux, llama.cpp/Ollama on macOS). Frontier (Claude/OpenAI) only via the Phase-5 policy gate, through the egress proxy.
- **Single-host deployment via OS-native user-level supervisor.** `systemd --user` (Linux) / `launchd` LaunchAgents (macOS). No k3s.
- **Fixed core tools, sandbox-bound agent-authored Python.** Critical workers are human-curated and shipped with the binary. Agent-authored code only runs inside `python-exec`'s strict sandbox; named/persisted skills get an optional human-approve gate (the L3 skill arc).
- **JSON-RPC 2.0 over stdio.** MCP-stdio compatible. Lets us swap in a richer MCP client later without changing the trust boundary.
- **Operator→daemon command channel = the Postgres `tasks` queue + `LISTEN/NOTIFY`** (not a new IPC socket). `ask` and `memory l3 run` both ride it; daemon-side execution against the single live `ToolRegistry` is the canonical pattern (#179 Opt-3).

## Inspirations / things to read before each milestone

Two adjacent OpenClaw-derived projects ship code we can read (Apache-2.0/MIT, AGPL-compatible) before each new milestone — convergent prior art saves design time:

- **ZeroClaw** ([`zeroclaw-labs/zeroclaw`](https://github.com/zeroclaw-labs/zeroclaw), 100% Rust): read [`crates/zeroclaw-runtime/src/security/`](https://github.com/zeroclaw-labs/zeroclaw/tree/main/crates/zeroclaw-runtime/src/security) — has working `bubblewrap.rs`, `landlock.rs`, `seatbelt.rs`, `firejail.rs`, `pairing.rs`, `webauthn.rs`, `leak_detector.rs`, `workspace_boundary.rs`. Architectural drawback vs us: tools run as in-process Rust traits, OS sandbox wraps the runtime — weaker boundary than our process-per-worker. Don't copy the in-process tool model.
- **IronClaw** ([`nearai/ironclaw`](https://github.com/nearai/ironclaw)): read its dispatcher chokepoint pattern (`ToolDispatcher::dispatch()` is the single audit/safety-validation funnel for *every* action, regardless of caller). Drawbacks: WASM-as-boundary is software-only containment; Postgres+libSQL dual backend is overkill at our stage.

The *defining* architectural difference: kastellan enforces **one OS process + one bwrap/Seatbelt jail per worker**. Both reference projects retreated from that. Don't.

## How to update this document at session end

**Header first, prose last.** The header is what the next session reads first
and treats as authoritative; stale header fields silently mislead future
sessions even when the prose is correct. Follow the steps in this order:

1. **Bump header fields at the top — before writing any prose:**
   - `Last updated:` → today's date.
   - **Current state / Last commit** → the hash of the most recent shipped commit. Confirm with `git log --oneline -1`.
   - `Session-end verification:` → re-run `cargo test --workspace` and copy the **passed / failed / ignored / `[SKIP]`** counts into this line.
   - **Every test-count number embedded elsewhere in the doc that changed this session** — a fresh agent grep-finds them and will trust whatever is there.
2. **Move "Next TODO" → "Recently completed (this session)"** if the picked option shipped, with enough detail (file paths, why-not-X, gotchas, test-count delta) that the next session can start cold.
3. **Write a fresh "Next TODO (pick one)"** with options sized for one session each — include file paths, gotchas, and the verification step.
4. **Refresh "Working state"** — anything new under stubs, anything that became real.
5. **Tick the matching items off in [`../ROADMAP.md`](../ROADMAP.md)** with the commit hash.
6. **Commit both files together** with a `docs(handover): ...` message.
7. **If a milestone shipped:** does `site/roadmap.html` (timeline + "Last
   updated" stamp, and the landing-page status numbers) need a one-line
   update? See `site/README.md`.

### Pruning convention

The handover should stay focused on **what the next session needs to act on**: the current state, the last 2–3 sessions in detail, and the next TODO. Older session entries get compressed into the "Earlier history" summary or dropped entirely once they're no longer load-bearing.

When HANDOVER.md grows past the point where the next session can absorb it cold (rough rule of thumb: more than a couple of screens of "Recently completed"), prune it:

1. **Snapshot first.** Copy the current HANDOVER.md to `archive/handover_<YYYYMMDD>[_<slug>].md` (e.g. `handover_20260605_pre-prune.md`). The archive is the audit trail — never edited after the fact, never deleted.
2. **Keep verbatim:** the header, "Read these first," "Working state" (current truth), the most recent 1–2 sessions of "Recently completed," "Key design decisions," "Next TODO," "Open follow-up issues," "Open questions," "Inspirations," and this section.
3. **Compress everything else** into a single "Earlier history" section: one bullet per session, naming the slice + the headline change + a pointer to the archive snapshot for full reasoning.
4. **Cross-link** from the compressed bullets to the archive snapshot so anyone who needs the full reasoning can find it.
5. **Commit the prune separately** with `docs(handover): prune older sessions, archive pre-prune snapshot` so the diff is reviewable.

The archive directory is the historical record; HANDOVER.md is the working brief.
