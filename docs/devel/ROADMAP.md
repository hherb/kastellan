# hhagent ‚Äî Development Roadmap

Sequenced feature list. Items are added whenever a feature decision is made
and ticked off as they ship. Order reflects expected build sequence ‚Äî earlier
items unlock later ones.

> **How to update.** When we agree on a new feature, append it to the most
> appropriate phase (or a new phase) ‚Äî placed at the position that respects
> dependencies. When an item ships, change `[ ]` ‚Üí `[x]` and add the commit
> hash. Don't delete completed items; they document the build sequence.

## Phase 0 cont. ‚Äî Audit log

- [x] Non-superuser `hhagent_runtime` role + DB-layer `REVOKE UPDATE, DELETE, TRUNCATE ON audit_log` so append-only is enforced by Postgres, not just by application discipline ‚Äî landed 2026-05-10 as Option L. New `db/migrations/0002_runtime_role.sql` creates the role (`NOSUPERUSER NOCREATEROLE NOCREATEDB NOLOGIN NOINHERIT`), grants the OS user (cluster bootstrap superuser) membership via `EXECUTE format('GRANT hhagent_runtime TO %I', current_user)`, and carves the GRANT/REVOKE shape: `audit_log` gets `SELECT, INSERT` only; the other five tables (`tasks`, `memories`, `entities`, `relations`, `secrets`) get full CRUD; sequences for all six get explicit `GRANT USAGE` (BIGSERIAL needs `nextval()`); `ALTER DEFAULT PRIVILEGES` covers future migrations' tables/sequences automatically. New `RUNTIME_ROLE` constant + `set_role_runtime_statement()` pure helper in `db/src/conn.rs` (paired regression-test pins). `db/src/probe.rs` now executes `SET ROLE hhagent_runtime` between `MIGRATOR.run` and the `audit_log` INSERT ‚Äî bootstrap identity (= OS user, peer auth) is preserved for migrations that need superuser (CREATE EXTENSION, CREATE ROLE), and the runtime role applies to every application write thereafter. New integration test `runtime_role_audit_log_revoke_is_enforced` proves the contract end-to-end: role exists with the expected `pg_roles` flags, OS user is recorded in `pg_auth_members`, INSERT under `SET ROLE` succeeds, UPDATE + DELETE both fail with `permission denied`, full CRUD on `memories` works, final `audit_log` count is exactly 2 (probe row + test INSERT, no UPDATE rewrite, no DELETE leak). 151 ‚Üí 154 tests on Linux (+2 db unit, +1 db integration). **Why `SET ROLE` over `pg_ident.conf` mapping**: keeps the change pure-SQL inside a sqlx migration, no need to mutate `pg_hba.conf`/`pg_ident.conf` post-`initdb`; the runtime-role privilege boundary is identical regardless of the entry route.
- [x] Append-only `audit_log` writer in core (every tool call, LLM call, channel I/O, memory write) ‚Äî landed 2026-05-10 as Option I (dispatcher write-site). `core::tool_host::dispatch(&PgPool, &mut SupervisedWorker, tool, method, params)` is the single chokepoint every Phase 0+ tool call goes through; one row per call (`actor = "tool:<name>"`, `action = <method>`, payload `{req, result|err, ms}`). Audit insert is best-effort (a transient DB issue must not mask a successful worker result), but the DB-layer durability story is strong: the `audit_log` REVOKE shape from Option L makes rows tamper-resistant, and the new `0003_audit_log_notify.sql` trigger emits `pg_notify('audit_log_inserted', NEW.id)` so the JSONL mirror is woken synchronously on every commit. Daemon-scoped `db::pool::connect_runtime_pool` opens a pool whose `after_connect` hook runs `SET ROLE hhagent_runtime` on every dialed connection ‚Äî closes issue #11 ahead of the Phase-1 timeline. **Channel I/O / LLM / memory writers** still need to plumb through `dispatch` when those tools materialise (Phase 2+ for channels, Phase 1 for memory), but the chokepoint pattern is the explicit contract now.
- [x] JSONL on-disk mirror under `~/.local/state/hhagent/audit-*.jsonl` (rotated) ‚Äî landed 2026-05-10 as Option I. `core::audit_mirror::spawn_mirror(pool, state_dir)` spawns a long-lived tokio task that holds a dedicated `PgListener` on `audit_log_inserted` and replicates committed rows to date-named files (`audit-YYYY-MM-DD.jsonl`, UTC). Initial drain via `audit::fetch_since(0)` so the bring-up row lands on cold starts; periodic 5 s catch-up SELECT in batches of 256 closes any NOTIFY-loss window from listener reconnects. fsync per write ‚Äî operator visibility beats throughput at Phase 0 scale. JSONL line shape: `{id, ts (RFC 3339), actor, action, payload}`, one row per line, single trailing newline. `MirrorHandle::shutdown` flips a cancellation watch and awaits the task before pool close so the final fsync always runs.
- [x] CLI viewer: `hhagent-cli audit tail` ‚Äî landed 2026-05-10 as Option I. New `[[bin]] hhagent-cli` in core/Cargo.toml; `core/src/audit_tail.rs` ships pure helpers (`parse_audit_filename`, `find_audit_files`, `tail_loop`) and the CLI binary wraps them. Subcommand `audit tail` supports `--from-start` (replay every line of every existing file), `--no-follow` (cat mode), and `--state-dir PATH` (override default). Polls 250 ms ‚Äî sub-second visibility without a directory-watch dep. SIGPIPE-style `BrokenPipe` maps to exit 0 (matches BSD `tail`). No DB connection required, so the viewer works against a daemon that has crashed mid-startup.
